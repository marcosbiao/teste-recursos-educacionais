Ideia central

Simplicidade é uma relação entre problema e solução

Uma solução é simples quando possui apenas os elementos necessários para resolver o problema de forma correta, compreensível e adequada. Código curto pode ser difícil de entender. Código um pouco maior pode ser mais claro e seguro.

Não pergunte apenas “qual tem menos linhas?”. Pergunte: qual explica melhor a solução sem trabalho desnecessário?

1

Preserve a correção

Uma simplificação não pode remover casos ou alterar o resultado.

2

Reduza o esforço mental

O leitor deve acompanhar os dados e as decisões sem reconstruir intenções ocultas.

3

Use a estrutura certa

A estratégia deve combinar com a quantidade de dados e com a ação solicitada.

Sinais de complicação desnecessária

Repetição manual

O mesmo comando aparece várias vezes quando um laço expressaria melhor o padrão.

Condições em excesso

Muitos testes repetem a mesma ideia ou criam caminhos difíceis de verificar.

Estruturas sem função

Vetores, matrizes ou funções aparecem, mas não resolvem uma necessidade do problema.

Otimização prematura

A solução fica menos legível para ganhar eficiência que o problema não exige.

Mesmo problema, duas estratégias

Problema: ler cinco números e apresentar a soma.
Estratégia A

Repetir comandos

  • Cinco variáveis diferentes.
  • Cinco leituras escritas manualmente.
  • Difícil ampliar para 20 valores.
Estratégia B

Usar laço e acumulador

  • Uma variável para o valor atual.
  • Uma regra de repetição.
  • Escala sem duplicar código.

Decisão: a estratégia B é mais adequada porque o problema contém uma ação repetida e não exige guardar os cinco valores.

estrategia_escolhida.c
#include <stdio.h>

int main() {
    int i;
    float valor, soma = 0;

    for (i = 0; i < 5; i++) {
        scanf("%f", &valor);
        soma += valor;
    }

    printf("Soma: %.2f\n", soma);
    return 0;
}

Exemplos progressivos

Desconto de 10%

Aplicar desconto somente quando a compra for igual ou superior a R$ 100.

Nível básico
Complicada

Cria variável lógica extra e dois testes para uma única decisão.

Mais simples

Testa diretamente a condição que altera o valor.

Avaliação: a condição direta preserva o significado e reduz estados intermediários.

desconto.c
#include <stdio.h>

int main() {
    float total;

    scanf("%f", &total);

    if (total >= 100.0) {
        total *= 0.90;
    }

    printf("Total: %.2f\n", total);
    return 0;
}

Soma de cinco valores

Ler cinco números e apresentar apenas a soma final.

Nível intermediário
  • Não é necessário guardar todos os valores.
  • O padrão de leitura e soma se repete cinco vezes.
  • Um laço torna a quantidade explícita e ajustável.
foracumuladorsem vetor

Avaliação: o laço reduz duplicação e o acumulador guarda somente a informação necessária.

soma.c
#include <stdio.h>

int main() {
    int i, valor, soma = 0;

    for (i = 0; i < 5; i++) {
        scanf("%d", &valor);
        soma += valor;
    }

    printf("%d\n", soma);
    return 0;
}

Maior valor de um vetor

Encontrar o maior elemento entre seis números já armazenados.

Nível intermediário
  • Comparações par a par geram muitos testes repetidos.
  • Um único percurso mantém o maior valor conhecido.
  • A inicialização usa o primeiro elemento, não um valor arbitrário.
vetorforif

Avaliação: um percurso linear é claro, escalável e suficiente para a tarefa.

maior.c
#include <stdio.h>

int main() {
    int valores[6] = {8, 3, 12, 5, 10, 7};
    int maior = valores[0];

    for (int i = 1; i < 6; i++) {
        if (valores[i] > maior) {
            maior = valores[i];
        }
    }

    printf("Maior: %d\n", maior);
    return 0;
}

Erros comuns

Confundir simples com curto

Por que acontece: o aluno avalia apenas a quantidade de linhas.

Como evitar: considere legibilidade, casos cobertos e facilidade de alteração.

Adicionar estruturas para parecer completo

Por que acontece: vetor, função ou matriz são usados sem necessidade.

Como evitar: associe cada estrutura a um requisito concreto.

Otimizar antes de compreender

Por que acontece: busca-se desempenho antes de garantir uma solução clara e correta.

Como evitar: primeiro construa e teste uma estratégia compreensível.

Remover etapas importantes

Por que acontece: simplificar é interpretado como apagar validações ou casos.

Como evitar: retire apenas o que não contribui para a correção ou compreensão.

Perguntas rápidas

A solução com menos linhas é sempre a mais simples?

Não. Menos linhas podem esconder etapas, combinar responsabilidades ou dificultar a leitura. A solução simples é a que resolve o problema com clareza e sem elementos desnecessários.

Usar uma função sempre torna o programa melhor?

Não. Uma função é útil quando separa uma responsabilidade clara, evita repetição ou melhora a leitura. Criar funções pequenas sem propósito pode aumentar a fragmentação.

Uma solução com dois percursos pode ser aceitável mesmo existindo uma solução com um percurso?

Sim. Para dados pequenos e em contexto introdutório, dois percursos claros podem ser adequados. A decisão deve considerar requisitos de desempenho, clareza e facilidade de verificação.
