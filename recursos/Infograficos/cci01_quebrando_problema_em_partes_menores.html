Ideia central

Decompor é tornar visível o raciocínio

Um problema grande pode ser explorado por diferentes pontos de entrada. Você pode começar pelos dados, pelas ações, pelas regras ou pelo resultado esperado. Depois, conecte essas partes em uma sequência coerente.

Não existe apenas um ponto de partida.
Problema completo

O que precisa ser resolvido?

Dados de entrada

Quais valores, textos ou informações o programa precisa receber?

Ações necessárias

Quais cálculos, comparações ou transformações precisam ocorrer?

Regras e condições

Há limites, critérios ou decisões que mudam o caminho da solução?

Resultado esperado

O que deve ser mostrado, armazenado ou informado ao final?

Fluxo para transformar partes em solução

Movimento 1

Ler e marcar

Destaque verbos, dados, restrições e o resultado solicitado no enunciado.

Movimento 2

Agrupar partes

Separe os elementos em entrada, processamento, regras e saída.

Movimento 3

Conectar relações

Verifique qual dado alimenta cada ação e qual ação produz cada resultado.

Movimento 4

Ordenar ações

Coloque os passos em uma sequência que possa ser executada sem lacunas.

Movimento 5

Validar

Teste mentalmente se as partes usam todos os dados e produzem a saída pedida.

↻

O fluxo pode voltar. Se uma etapa estiver vaga, retorne ao mapa, divida essa etapa novamente e torne a ação mais específica.

Código curto em C

Uma sequência linear
#include <stdio.h>

int main() {
    float nota1, nota2, media;

    scanf("%f", &nota1);
    scanf("%f", &nota2);

    media = (nota1 + nota2) / 2;

    printf("Media: %.2f\n", media);
    return 0;
}

Uma entrada, duas transformações
#include <stdio.h>

int main() {
    int totalMinutos, horas, minutos;

    scanf("%d", &totalMinutos);

    horas = totalMinutos / 60;
    minutos = totalMinutos % 60;

    printf("%d h e %d min\n", horas, minutos);
    return 0;
}

Código curto em C

Fluxo com decisão
#include <stdio.h>

int main() {
    float n1, n2, n3, media;

    scanf("%f %f %f", &n1, &n2, &n3);
    media = (n1 + n2 + n3) / 3;

    printf("Media: %.2f\n", media);

    if (media >= 7)
        printf("Aprovado\n");
    else
        printf("Recuperacao\n");

    return 0;
}

Erros comuns

Começar pelo código

Por que acontece: o aluno tenta resolver digitando comandos antes de compreender as partes do problema.

Como evitar: represente primeiro dados, ações, regras e saída.

Criar etapas muito amplas

Por que acontece: expressões como “resolver o cálculo” escondem várias ações menores.

Como evitar: use verbos específicos, como ler, somar, dividir, comparar e mostrar.

Esquecer as relações

Por que acontece: as partes são listadas, mas não fica claro qual dado alimenta cada operação.

Como evitar: conecte cada entrada à ação que a utiliza e cada ação ao resultado produzido.

Perguntas rápidas

É obrigatório começar a decomposição pelas entradas?

Não. Você pode começar pelo resultado esperado, pelas regras ou pelas ações. O importante é identificar todas as partes e conectá-las de forma coerente.

Como saber se uma etapa ainda está grande demais?

Se a etapa contém várias ações ou usa um verbo genérico, ela provavelmente pode ser dividida em passos mais específicos.

Qual é a diferença entre decompor e apenas listar comandos?

Decompor organiza o problema conceitualmente. A lista de comandos aparece depois, quando as partes já estão relacionadas e ordenadas.

Fechamento pedagógico

Depois de estudar este infográfico, o aluno deve conseguir explorar um problema por diferentes pontos de entrada, separar seus componentes e conectá-los em uma solução algorítmica clara antes da implementação em C.
