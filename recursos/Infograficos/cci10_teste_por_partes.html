Ideia central

Um erro é mais fácil de localizar quando o teste é pequeno

Se o programa inteiro falhar, várias partes podem ser responsáveis. Ao testar cada bloco com entradas controladas e um resultado esperado, você reduz o espaço de investigação.

Princípio: uma parte só deve ser integrada quando produzir o resultado previsto nos casos escolhidos.
1

Isole

Execute uma função, um laço ou uma consulta sem depender do restante do sistema.

2

Preveja

Defina o resultado correto antes de executar. Sem previsão, não há comparação confiável.

3

Integre

Una partes já verificadas e teste também a comunicação entre elas.

Roteiro de teste por partes

1. Escolha a parte

Defina exatamente qual função, laço, consulta ou saída será observada.

2. Fixe a entrada

Use valores pequenos, como {3, 5, 2}, fáceis de calcular manualmente.

3. Preveja

Escreva o resultado esperado antes da execução.

4. Execute e compare

Registre o resultado observado e identifique qualquer diferença.

5. Integre

Conecte a parte aprovada ao restante e teste o fluxo completo.

Exemplos progressivos

Exemplo 1: testar uma função sem usar scanf

Nível básico
Objetivo

Verificar se a função dobro devolve o valor correto para entradas simples.

Plano de teste
Entrada 14 → esperado 8
Entrada 20 → esperado 0
Entrada 3-3 → esperado -6
Por que funciona?

Os valores são escritos diretamente no código. Assim, problemas de leitura não interferem no teste da função.

teste_dobro.c
#include <stdio.h>

int dobro(int valor) {
    return valor * 2;
}

int main() {
    printf("%d\n", dobro(4));
    printf("%d\n", dobro(0));
    printf("%d\n", dobro(-3));

    // Saída esperada: 8, 0 e -6
    return 0;
}

Exemplo 2: testar uma consulta encontrada e ausente

Nível intermediário
Objetivo

Confirmar que a busca devolve a posição correta e também sinaliza quando o código não existe.

Plano de teste
Caso ABuscar 20 → esperado índice 1
Caso BBuscar 99 → esperado -1
LimiteBuscar 30 → esperado índice 2
Evidência

Uma única busca bem-sucedida não demonstra o comportamento do caso ausente.

teste_busca.c
#include <stdio.h>

int buscarCodigo(int codigos[], int n, int alvo) {
    for (int i = 0; i < n; i++) {
        if (codigos[i] == alvo) {
            return i;
        }
    }
    return -1;
}

int main() {
    int codigos[3] = {10, 20, 30};

    printf("%d\n", buscarCodigo(codigos, 3, 20));
    printf("%d\n", buscarCodigo(codigos, 3, 99));
    printf("%d\n", buscarCodigo(codigos, 3, 30));

    // Saída esperada: 1, -1 e 2
    return 0;
}

O que está sendo testado agora?

Não apenas as funções isoladas, mas o envio dos dados e o uso correto dos retornos no main.

teste_integracao.c
#include <stdio.h>

int somarEstoque(int v[], int n) {
    int total = 0;
    for (int i = 0; i < n; i++) {
        total += v[i];
    }
    return total;
}

int buscarCodigo(int v[], int n, int alvo) {
    for (int i = 0; i < n; i++) {
        if (v[i] == alvo) return i;
    }
    return -1;
}

int main() {
    int codigos[3] = {10, 20, 30};
    int estoque[3] = {3, 5, 2};
    int pos = buscarCodigo(codigos, 3, 20);

    printf("Total: %d\n", somarEstoque(estoque, 3));
    if (pos != -1) {
        printf("Estoque do codigo 20: %d\n", estoque[pos]);
    }

    // Esperado: Total 10 e estoque 5
    return 0;
}

Erros comuns

Testar somente o programa inteiro

Por que acontece: parece mais rápido executar tudo de uma vez.

Como evitar: verifique primeiro os blocos que produzem resultados próprios.

Não definir o esperado

Por que acontece: o aluno olha a saída e decide intuitivamente se parece correta.

Como evitar: calcule manualmente o resultado antes da execução.

Usar apenas um caso comum

Por que acontece: um resultado correto gera confiança excessiva.

Como evitar: inclua zero, limites, ausência e valores contrastantes.

Alterar várias partes ao mesmo tempo

Por que acontece: várias hipóteses são corrigidas em uma única tentativa.

Como evitar: mude uma parte, execute novamente e registre o efeito.

Perguntas rápidas

Por que usar valores pequenos ao testar um laço?

Porque eles permitem acompanhar manualmente cada iteração, o índice e a mudança das variáveis.

Uma consulta foi testada apenas com um código existente. O teste está completo?

Não. Também é necessário testar um código ausente para verificar o valor usado para indicar que nada foi encontrado.

Quando as partes devem ser integradas?

Depois que cada parte produzir o resultado esperado nos casos escolhidos. Em seguida, testa-se a comunicação entre elas e o fluxo completo.

Fechamento pedagógico

Depois de estudar este infográfico, o aluno deve conseguir planejar testes pequenos para funções, laços, consultas e saídas, registrar resultados e integrar somente as partes que já foram verificadas.

Teste de função

Uma função pode ser verificada com argumentos fixos, sem depender da leitura do teclado ou da saída final do programa.

Mesmo exemplo: programa de estoque
ChamadasomarEstoque({3,5,2}, 3)
Retorno esperado10
Regra prática: escolha entradas cujo resultado você consiga calcular manualmente. Compare diretamente o retorno da função.
×

Teste de laço

O teste deve observar o valor inicial, a condição, a atualização e o estado produzido em cada repetição.

Mesmo exemplo: programa de estoque
Índices esperados0, 1 e 2
Totais parciais3 → 8 → 10
Regra prática: use uma coleção curta e anote uma linha para cada iteração. Isso revela limites incorretos e atualizações ausentes.
×

Teste de consulta

Uma consulta precisa demonstrar tanto o caminho em que encontra um elemento quanto o caminho em que percorre os dados sem encontrar.

Mesmo exemplo: programa de estoque
Caso encontradobuscar 20 → posição 1
Caso ausentebuscar 99 → -1
Regra prática: teste o primeiro elemento, o último, um elemento intermediário e um valor inexistente quando a consulta for importante.
×

Teste da saída

A lógica pode estar correta, mas a apresentação ainda pode ter valor, ordem, rótulo ou formatação incorretos.

Mesmo exemplo: programa de estoque
EsperadoTotal em estoque: 10
Comparartexto, espaço, valor e quebra de linha
Regra prática: trate a saída esperada como uma especificação. Não avalie apenas se o número “parece certo”.
×

Teste de integração

Mesmo partes corretas podem falhar quando um retorno é ignorado, um tamanho errado é enviado ou os dados são ligados na ordem incorreta.

Mesmo exemplo: programa de estoque
Fluxobuscar posição → acessar estoque[pos]
Saída esperadaTotal 10 | código 20: 5
Regra prática: no teste do todo, concentre-se primeiro nas fronteiras entre as partes: argumentos, retornos, índices e ordem das chamadas.
