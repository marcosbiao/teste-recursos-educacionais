Correção local sem testar o conjunto

Painel diagnóstico para reconhecer, compreender e corrigir alterações que resolvem uma parte do programa, mas comprometem o fluxo completo.

Gravidade

Médio

O erro pode passar despercebido em testes parciais.

Sintoma observado

O aluno corrige uma parte, mas cria erro em outra.

Impacto no programa

A solução parece funcionar no trecho alterado, mas o resultado final deixa de representar corretamente todo o processamento.

Referência diagnóstica

Enunciado analisado

Use este enunciado como base para comparar o que foi pedido com o que o código realmente faz depois da correção local.

Leia as vendas de 5 dias, some todas as vendas informadas e mostre o total final.

Analise antes de ver a explicação

#include <stdio.h>

int main() {
    float venda, total = 0;

    for (int dia = 1; dia < 5; dia++) {
        printf("Venda do dia %d: ", dia);

    return 0;
}

Origem do problema

Causa provável

Esse erro costuma ocorrer quando o aluno ajusta apenas o ponto em que percebeu o problema, sem verificar se a mudança continua coerente com as outras estruturas conectadas ao programa.

Sinais observáveis

Evidência no código

  • Padrão detectável: a condição do laço foi alterada, mas o cálculo final continua dependendo de todas as repetições.
  • Comportamento observado: o programa executa sem erro de sintaxe, mas deixa uma venda fora do total.
  • Teste revelador: informar 5 valores simples e conferir se todos participaram da soma.

Contexto típico

Quando aparece

Quando uma solução envolve várias estruturas conectadas, como leitura em laço, atualização de acumulador e saída final. Uma mudança local no laço pode quebrar o cálculo global.

Código com erro

Exemplo com erro

#include <stdio.h>

int main() {
    float venda, total = 0;

    for (int dia = 1; dia < 5; dia++) { // erro: lê apenas 4 dias
        printf("Venda do dia %d: ", dia);
        scanf("%f", &venda);
        total = total + venda;
    }

    printf("Total: %.2f\n", total);
    return 0;
}

O enunciado pede 5 vendas, mas a condição dia < 5 faz o laço executar apenas para os dias 1, 2, 3 e 4. A correção local na condição do laço quebrou o cálculo final, pois o total deixa de incluir o quinto dia.

Código corrigido

Correção mínima

#include <stdio.h>

int main() {
    float venda, total = 0;

    for (int dia = 1; dia <= 5; dia++) { // correção: lê os 5 dias
        printf("Venda do dia %d: ", dia);
        scanf("%f", &venda);
        total = total + venda;
    }

    printf("Total: %.2f\n", total);
    return 0;
}

A correção muda apenas a condição do laço para incluir o dia 5. Agora a leitura, o acumulador e a saída final permanecem coerentes com o enunciado.

Contraste essencial

Antes e depois

Compare a lógica incorreta com a lógica corrigida. A diferença principal está em testar o fluxo completo, não apenas o trecho modificado.

Lógica incorreta

para dia de 1 enquanto dia < 5:
    ler venda
    somar venda no total

mostrar total

A leitura foi reduzida a 4 repetições, mas a saída continua aparentando ser o total final.

Lógica correta

para dia de 1 enquanto dia <= 5:
    ler venda
    somar venda no total

mostrar total

A condição percorre todos os 5 dias, preservando a relação entre entrada, processamento e saída.

Conceito

Por que isso acontece

Em soluções integradas, mudanças locais podem afetar o comportamento global. Um ajuste no laço, por exemplo, altera também quantos dados entram no cálculo final.

Estratégia prática

Como identificar

Teste casos completos, não apenas trechos isolados. Para este caso, use entradas simples como 10, 20, 30, 40 e 50. O total esperado é 150. Se o programa mostrar 100, alguma etapa ficou fora do fluxo.

Princípio de correção

Forma correta

Executar testes que passem por todas as etapas principais do programa.

  1. Revise o enunciado completo.
  2. Identifique entrada, processamento e saída.
  3. Aplique a alteração mínima necessária.
  4. Teste o fluxo inteiro com dados conhecidos.

Autoavaliação curta

Perguntas de verificação

Por que testar apenas o trecho alterado pode ser insuficiente?

Porque o trecho alterado pode estar conectado a outras partes do programa. A mudança pode parecer correta localmente, mas produzir uma saída final incoerente.

Qual sinal indica uma correção local sem teste do conjunto?

Um sinal comum é o programa continuar executando, mas deixar de processar todos os dados exigidos pelo enunciado.

O que deve ser feito depois de alterar a condição de um laço?

Executar um teste completo, verificando se todas as entradas esperadas foram lidas, processadas e refletidas na saída final.

Aplicação diagnóstica

Teste rápido

Mini enunciado: leia 5 números inteiros, conte quantos são pares e mostre a quantidade final.
#include <stdio.h>

int main() {
    int numero, pares = 0;

    for (int i = 1; i < 5; i++) { // alteração local deixou um número fora
        scanf("%d", &numero);
        if (numero % 2 == 0) {
            pares++;
        }
    }

    printf("Pares: %d\n", pares);
    return 0;
}
